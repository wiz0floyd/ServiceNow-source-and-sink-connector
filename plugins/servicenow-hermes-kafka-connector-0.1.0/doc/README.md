# ServiceNow Hermes Kafka Connector

Kafka Connect sink connector that streams data from Confluent Cloud into a ServiceNow instance Hermes Kafka cluster over mutual TLS (mTLS).

## What it does

- Reads from a Confluent Cloud topic (any data — pass-through bytes).
- Writes to a configured Hermes topic on your ServiceNow instance.
- Authenticates via mTLS using a certificate generated by the ServiceNow Instance PKI Certificate Generator.
- Preserves all Kafka record headers verbatim.
- Fails loud at startup if the Hermes topic does not exist (no auto-create).

## Why it exists

Hermes is the Kafka cluster that fronts ServiceNow's Stream Connect data plane. Confluent's Cluster Linking does not solve the Hermes-as-destination direction. This connector ships as a Confluent Cloud Custom Connector so customers can move data from Confluent Cloud into ServiceNow workflows without VPN, PrivateLink, or a self-hosted Connect cluster.

## Requirements

- ServiceNow instance with the Hermes Messaging Service activated
- An mTLS certificate generated via **Instance PKI Certificate Generator** (PKCS12 keystore and truststore)
- Confluent Cloud cluster on a public-internet or PrivateLink-enabled endpoint

## Configuration

| Property | Required | Description |
|---|---|---|
| `hermes.instance.name` | yes | Bare instance name, e.g. `myinstance` (drives `myinstance.service-now.com:4000-4003` bootstrap) |
| `hermes.topic` | yes | Full Hermes topic name, e.g. `snc.myinstance.sn_streamconnect.my-topic` |
| `hermes.ssl.keystore.b64` | yes | Base64-encoded PKCS12 keystore from Instance PKI Certificate Generator |
| `hermes.ssl.keystore.password` | yes | Keystore password |
| `hermes.ssl.truststore.b64` | yes | Base64-encoded PKCS12 truststore |
| `hermes.ssl.truststore.password` | yes | Truststore password |
| `hermes.producer.acks` | no | Default `all` |
| `hermes.producer.retries` | no | Default `Integer.MAX_VALUE` |

Standard Connect properties (`topics`, `tasks.max`, `key.converter`, `value.converter`) apply.

## Encoding the keystore

After downloading the keystore from Instance PKI Certificate Generator:

```bash
# Linux / macOS
base64 -i keystore.p12

# Windows PowerShell
[Convert]::ToBase64String([IO.File]::ReadAllBytes("keystore.p12"))
```

## Delivery semantics

End-to-end **at-least-once**. The producer is configured with `enable.idempotence=true` and `acks=all`. Idempotency / dedup is the consumer's responsibility on the Hermes side.

## Sensitive properties

All four mTLS-related configs are declared `Type.PASSWORD` in the connector's `ConfigDef`, which causes Kafka Connect to mask them in REST and log output. When uploading to Confluent Cloud, also pass them via the platform-level sensitive list so they are encrypted at rest:

```
hermes.ssl.keystore.b64,hermes.ssl.keystore.password,hermes.ssl.truststore.b64,hermes.ssl.truststore.password
```

### CLI upload

```bash
confluent connect custom-plugin create hermes-sink-connector \
  --plugin-file target/components/packages/servicenow-hermes-kafka-connector-0.1.0.zip \
  --connector-type sink \
  --connector-class com.servicenow.kafka.connect.hermes.HermesSinkConnector \
  --sensitive-properties hermes.ssl.keystore.b64,hermes.ssl.keystore.password,hermes.ssl.truststore.b64,hermes.ssl.truststore.password \
  --cloud aws
```

### Console upload

In the Confluent Cloud Console **Add Plugin** flow, paste the four property names into the **Sensitive properties** field when prompted.

The canonical list is exported from the code at `HermesConnectorConfig.SENSITIVE_PROPERTIES_CSV` — if more sensitive configs are added later, update that constant and they'll appear here automatically.
